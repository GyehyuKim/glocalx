## 1. User Interview (P01)

35세 직장인, 민음사 빅팬에 듀오링고 1년 이상 streak 유지자. 책은 잘 사두지만 펴지를 못 하는 사람. 작년 24권 목표에서 14권에서 멈췄고 본인은 "겁난다"고 표현했다.

**(a) Biggest lesson.** ReadingGo 의 핵심 차별점이라고 생각했던 *함께 읽기* 가 정작 타겟 사용자에게는 *진입장벽* 으로 작동했다. 4–6명이 같은 책을 읽는 구조가 "내가 안 읽으면 다른 분들한테 미안해질 것 같다" 는 회피 트리거가 되어 오히려 앱을 안 열게 만든다. 가설은 사회적 동기가 습관 형성을 돕는다였으나, *시작이 어려운* 사용자에게는 사회적 압력이 시작을 더 막는다. 그리고 더 깊이 들어가니 진짜 통증은 진도 자체가 아니라 *정체성과 행동의 간격* 이었다. "민음사 신간을 챙겨보는 나" 와 "그 책을 안 펴는 나" 사이의 거리를 좁히고 싶은 욕구가 진짜 leaf 였다. 가치 제안의 축이 진도 트래킹이 아니라 정체성 회복일 수 있다.

**(b) One thing to fix.** 온보딩에서 함께 읽기를 옵션으로 빼고, "오늘 N쪽 + streak" 만으로 2–4주 혼자 굴러가게 한 다음 함께 읽기를 단계적으로 제안하는 흐름으로 바꿀 것. P01 본인도 "혼자 쓰는 모드만 있으면 바로 깔 것 같아요. 두세 달 혼자 잘 굴러간 다음에 천천히 붙여 보고 싶어요" 라고 명시했다.

**(c) Commit SHA.** `c67d600` (브랜치 `gyehyu/interview-p01` head) → `b326d78` (PR #111, main 머지). 파일: `docs/4. user_interviews/2026-05-28_P01.md`.

## 2. /health 와 test infrastructure

먼저 spec ↔ 구현 정합 검증을 위한 python verifier 두 개(`tests/spec-align/nest.py`, `tests/spec-align/drift.py`)를 Ralph 스캐폴딩(PR #113, `40e485c` → main `247c403`)으로 깔았고, 그 위에 `/health` 가 의미 있게 굴러가도록 `markdownlint-cli2` 와 spec-align verifier 를 `CLAUDE.md` 의 `## Health Stack` 섹션에 등록했다(PR #116, `7710e8d` → main `e04cac9`). 첫 실행 결과는 6 카테고리 중 2개만 채워졌다(lint-md, test-spec-align). composite 10/10 이지만 표면이 좁다는 정직한 한계를 동반한다.

**(a) Major lesson.** `/health` 의 점수는 *코드가 건강하다* 가 아니라 *측정 도구가 있다* 를 말한다. 우리 프로젝트가 정적 HTML + JS 기반의 Phase 0 라서 `/health` 의 자동 감지 대상인 `tsc / biome / eslint / knip / shellcheck` 가 아예 없다. 흥미로운 점은 HTML 자체에 대한 린터가 `/health` 의 감지 목록에 *애초부터 빠져 있다* 는 사실이었다. AI 도구가 가정하는 "web stack" 은 빌드 도구가 박힌 React/TypeScript 환경이지, CDN 기반의 정적 셋업은 측정 대상으로 인지조차 못 한다. 즉 *빌드 도구가 곧 측정 표면* 이다. 그래서 `package.json` + ESLint 도입(2–3시간, Stack Lock 룰 영향)과 최소 침습(spec-align verifier + 마크다운 린터만 등록, 30분–1시간) 중 후자를 택했다 — 지금 Vite/TypeScript 로 옮기는 비용을 치를 시점이 아니라서. 진짜 가치는 점수 자체가 아니라 *시계열* 에 있다. `health-history.jsonl` 에 매 실행마다 한 줄이 쌓이고, 다음 실행부터 trend 가 보인다. 어떤 PR 이 score 를 떨어뜨렸는지, 어떤 머지 이후 회귀가 생겼는지 데이터로 추적 가능. 첫 baseline 의 의미는 *지금 잘하고 있다* 가 아니라 *기준선을 잡았다* 다.

## 3. oh-my-claudecode 루프 구현

ReadingGo 의 둥지 진화 마이크로카피 4종(spec §5.2 의 "참새가 자리를 잡았어요!" 등)을 코드에 정합시키는 작업을 Ralph 루프에 태웠다. 분할본 spec 을 source of truth 로 보고, 먼저 autopilot 으로 spec ↔ 구현 gap 분석을 돌려 67 항목 매트릭스(PR #115, `a9726cd`)를 산출했다. UX 기준으로 후보 3개 중 *첫 7일 가속 + 진화 카피* 를 골랐고, Ralph 가 단일 iteration 으로 13개 invariant 를 통과시켰다. 변경은 `data.js` +15 lines(PR #114, `99abf9b` → main `5951db3`). PRD → 구현 → architect 검증 → deslop → regression 의 전 단계를 무인으로 돌렸다.

**(a) Biggest lesson.** 가장 오래 걸린 일은 코드를 짜는 것이 아니라 *무엇을 루프에 태울지 정하는 일* 이었다. Ralph 자체는 단일 iteration 으로 verifier 를 통과했지만, 그 PROMPT 가 잘 정의되기까지 다섯 번의 의사결정 지점을 거쳤다. 처음 무심코 `/health` 를 돌렸을 때 AI 가 "측정할 표면이 0" 이라고 정직하게 보고해서 우선순위를 다시 잡았고, monolith spec 과 분할본이 공존하는 상황에서 *분할본을 원본으로 본다* 는 결정을 직접 박았다. "스펙 다 구현" 을 autopilot 에 곧장 던지려 하자 AI 가 3,831 LOC 기존 구현을 덮어쓸 위험을 경고했고, 그래서 gap 분석을 먼저 했다. 후보 셋 중 UX 기준으로 어느 게 가장 유효한지 물었더니 AI 가 spec §5.2 에 "첫 일주일에 시각적 변화가 자주 터져야 사용자가 '내 행동이 집을 짓고 있다' 라고 느낀다" 라고 *UX 의도가 명시* 되어 있다는 점을 짚었고, 그 근거로 첫 7일 가속 항목을 골랐다. 특히 인상 깊었던 순간은 AI 가 *틀린 자동 추종* 을 했을 때다. 작업 도중 "이제 `/health` 가시죠?" 라고 안내했는데, 내가 "아직 구현도 안 했는데 무슨 health?" 라고 막았다. AI 의 자율성은 강력하지만, *맥락이 바뀌었을 때 과거의 약속을 그대로 추종하지 않게* 사람이 끊어줘야 한다. PR 머지 권한을 GitHub 웹에서 사람이 행사하도록 분리한 것도 같은 맥락의 안전핀이다. 결론: Ralph 는 의사결정을 자동화하지 않는다. 오히려 *의사결정을 강제로 앞당겨* 잘 정의된 PROMPT 에 박게 만든다. 자율 루프의 진짜 가치는 *사람의 결정 단계를 더 진지하게 다루게 만드는 압력* 에 있다.

## 종합

세 과제를 관통하는 학습은 *AI 에게 무엇을 맡기지 않을지* 를 결정하는 일이 *맡기는 일* 보다 어렵다는 점이었다. 사용자 인터뷰에서는 "함께 읽기" 가 차별점인지 진입장벽인지 본인의 가설을 의심해야 했고, `/health` 에서는 빌드 도구가 없는 상황에서 무엇을 측정 가치로 볼지 정의해야 했고, Ralph 에서는 다섯 번의 의사결정 지점이 있었다. AI 가 보고하고 사람이 결정하는 분업이 명확해질수록 자율 도구의 결과가 *제품에 의미 있는* 방향으로 수렴한다는 것이 이번 주의 가장 큰 배움이다.
